# 회원 계보 검색 앱 기획안

## 1. 앱 목적

엄마가 회원번호를 기준으로 회원을 빠르게 찾고, 해당 회원이 누구의 왼쪽·오른쪽에 연결되어 있는지 확인하는 내부용 앱이다.

핵심 사용 흐름:

1. 회원번호·이름·닉네임·전화번호로 회원 검색
2. 회원 상세정보 확인
3. 상위 회원과 좌우 관계 확인
4. 상위·하위 회원으로 이동
5. 기준 회원으로부터의 계보 위치 자동 계산
6. 회원 추가·수정·숨기기
7. 주라인 사업자만 따로 보기

사용자는 J님과 엄마 두 명으로 제한한다.

---

## 2. MVP 범위

### 포함

- Supabase 로그인
- 회원 검색
  - 회원번호
  - 가입 당시 이름
  - 닉네임
  - 전화번호
- 회원 상세 화면
- 상위 회원 표시
- 좌·우 관계 표시
- 관계 이동
  - 상위 회원으로 이동
  - 직접 연결된 하위 회원 목록
  - 하위 회원 상세 화면으로 이동
- 기준 회원으로부터의 위치 자동 계산
  - 예: 우좌, 우우
- 주라인 사업자 표시 및 필터
- 회원 추가
- 회원 수정
- 회원 숨기기
- 미연결 회원 필터
- 탈퇴 회원 표시
- 앱 내 실제 삭제 금지

### 보류

엄마에게 필요 여부를 확인한 뒤 추가한다.

- 최근 검색
- 회원번호 기준 이전·다음 회원
- 전체 계보도 시각화
- 여러 닉네임 또는 aliases 관리

## 3. 탈퇴 회원 처리

최초 단계부터 탈퇴 상태를 저장하는 열을 추가한다.

권장 열:

```text
member_status
```

허용값:

```text
active
withdrawn
unknown
```

이유:

- 탈퇴 회원도 기존 계보 연결을 유지할 수 있다.
- 탈퇴 회원을 삭제하면 그 아래 회원들의 관계 추적이 끊길 수 있다.
- 검색 결과에서 제외하거나 `탈퇴` 배지를 붙이는 식으로 처리할 수 있다.
- 상태를 모르는 기존 자료는 `unknown`으로 이전할 수 있다.

`is_withdrawn` boolean보다 `member_status`를 권장한다. 향후 휴면·정지 같은 상태가 필요해져도 확장하기 쉽다.

## 4. 데이터 원칙

### 이름

```text
name
```

애터미 가입 당시 이름을 저장한다.

예:

```text
Demo Member Alpha
```

### 닉네임

```text
nickname
```

엄마가 실제로 검색하고 기억하는 이름을 저장한다.

예:

```text
테스트 알파
```

닉네임은 최소 두 글자로 입력한다. aliases 기능은 MVP에서 제외한다.

### 기존 위치 메모

```text
sponsor_name_raw
```

기존 시트의 원문을 보존한다.

예:

```text
테스트
테스트 알파
테스트 소속
```

이 열은 최초 데이터 연결 작업과 오류 확인에 사용한다. 관계 연결이 끝난 뒤에도 원본 확인용으로 남긴다.

### 실제 관계 연결

```text
sponsor_id
side
```

- `sponsor_id`: 같은 members 테이블에 있는 상위 회원의 내부 ID
- `side`: `left`, `right`, `unknown`

예:

```text
name = 새 회원
sponsor_id = 테스트 알파 회원의 내부 ID
side = left
```

화면에서는 `테스트 알파 / 좌`로 보이지만, 데이터베이스에는 이름이 아니라 고유 ID가 저장된다.

## 5. Supabase 테이블

테이블 이름:

```text
members
```

권장 열:

| 열 | 타입 | 설명 |
|---|---|---|
| `id` | uuid | Supabase 내부 고유 ID |
| `member_number` | text | 애터미 회원번호 |
| `name` | text | 가입 당시 이름 |
| `nickname` | text nullable | 엄마가 사용하는 이름 |
| `is_mainline` | boolean | 주라인 사업자 여부 |
| `sponsor_id` | uuid nullable | 상위 회원의 `members.id` |
| `side` | text nullable | `left`, `right`, `unknown` |
| `sponsor_name_raw` | text nullable | 기존 시트 원문 |
| `birth_date` | date nullable | 생년월일 |
| `phone` | text nullable | 숫자로만 저장한 전화번호 |
| `country_code` | text nullable | `KR`, `BR` 등 |
| `cpf` | text nullable | CPF |
| `notes` | text nullable | 자유 메모 |
| `member_status` | text | `active`, `withdrawn`, `unknown` |
| `is_hidden` | boolean | 앱에서 숨김 여부 |
| `created_at` | timestamptz | 생성 시각 |
| `updated_at` | timestamptz | 수정 시각 |

### 삭제 정책

앱에는 삭제 버튼을 만들지 않는다.

- 일반 사용: 수정 또는 숨기기만 가능
- 실제 삭제: J님이 Supabase Dashboard에서만 수행
- 기본 검색: `is_hidden = false`
- 숨김 회원 관리: J님 전용 화면 또는 Supabase Dashboard

## 6. 현재 CSV 상태와 추가할 열

현재 샘플 CSV에 있는 열:

```text
member_number
name
nickname
is_mainline
sponsor_name_raw
side
birth_date
phone
country_code
cpf
notes
```

추가 권장 열:

```text
member_status
is_hidden
```

초기값:

```text
member_status = unknown
is_hidden = FALSE
```

`sponsor_id`는 CSV에서 직접 만들 필요가 없다. 먼저 회원들을 Supabase에 등록한 뒤, `sponsor_name_raw`와 닉네임을 바탕으로 연결한다.

## 7. 기존 데이터 이전 방식

### 1단계: 회원 기본정보 등록

CSV의 모든 회원을 먼저 `members`에 등록한다.

이 단계에서는 다음 값이 비어 있어도 된다.

```text
sponsor_id
side
birth_date
phone
nickname
```

### 2단계: 상위 회원 후보 찾기

각 회원의 `sponsor_name_raw`를 기준으로 다음 열에서 상위 회원을 검색한다.

1. `nickname` 정확히 일치
2. `name` 정확히 일치
3. 일치 결과가 한 명일 때만 자동 연결

### 3단계: 연결 결과 분류

- 정확히 한 명 일치: 자동 연결
- 두 명 이상 일치: 확인 필요
- 일치 없음: 확인 필요
- `side` 없음: 좌우 확인 필요

### 4단계: 앱에서 수동 연결

미연결 회원 상세 화면:

```text
상위 회원: 미연결
기존 메모: 테스트 소속 / 좌
[상위 회원 연결하기]
```

엄마가 상위 회원 검색 결과에서 실제 사람을 선택하면 `sponsor_id`가 저장된다.

## 8. 검색

하나의 검색창에서 다음을 모두 검색한다.

- 회원번호
- 가입 당시 이름
- 닉네임
- 전화번호

검색 우선순위:

1. 회원번호 완전 일치
2. 회원번호 앞부분 일치
3. 닉네임 완전 일치
4. 이름·닉네임 포함
5. 전화번호 숫자 포함

검색 결과 예시:

```text
10000001
Demo Member Alpha
테스트 알파
주라인
```

## 9. 화면 구성

### 로그인 화면

J님과 엄마 계정만 로그인한다.

### 홈 화면

- 큰 통합 검색창
- 주라인 사업자 버튼
- 미연결 회원 버튼
- 회원 추가 버튼

추후 검토:

- 최근 검색
- 이전·다음 회원

### 회원 상세 화면

표시 정보:

- 이름
- 닉네임
- 회원번호
- 주라인 여부
- 회원 상태
- 생년월일
- 전화번호
- 국가
- CPF
- 메모
- 상위 회원
- 상위 회원 기준 좌·우
- 기준 회원으로부터 계산된 위치

관계 이동:

- 상위 회원 보기
- 왼쪽 직속 회원 보기
- 오른쪽 직속 회원 보기
- 직접 연결된 전체 회원 보기

관리 기능:

- 수정
- 숨기기
- 상위 회원 연결 또는 변경

### 주라인 화면

`is_mainline = true`인 회원만 표시한다.

- 이름
- 닉네임
- 회원번호
- 현재 상태
- 상세 화면 이동

### 미연결 회원 화면

다음 항목을 모아 보여준다.

- `sponsor_id` 없음
- `side` 없음
- 기존 메모와 일치하는 회원 없음
- 후보가 두 명 이상
- 탈퇴 여부 확인 필요

## 10. 계보 위치 계산

각 회원은 상위 회원 한 명과 좌·우 한 방향만 저장한다.

예:

```text
기준 회원 A
└─ 우: 회원 B
   ├─ 좌: 회원 C
   └─ 우: 회원 D
```

앱이 상위 관계를 거슬러 올라가 위치 문자열을 계산한다.

```text
회원 B = 우
회원 C = 우좌
회원 D = 우우
```

특정 기준 회원을 선택하면 해당 기준으로 다시 계산할 수 있다.

관계가 중간에서 끊긴 경우:

```text
위치 계산 불가
상위 관계 확인 필요
```

로 표시한다.

## 11. 신규 회원 입력

엄마가 내부 ID를 직접 입력할 일은 없다.

입력 화면:

```text
회원번호
가입 이름
닉네임
생년월일
전화번호
국가
상위 회원 검색
좌 / 우
주라인 여부
회원 상태
메모
```

상위 회원 입력 흐름:

1. 닉네임 또는 회원번호 검색
2. 검색 결과에서 실제 회원 선택
3. 좌·우 선택
4. 저장

화면에는 사람 이름이 보이지만 Supabase에는 선택한 회원의 `id`가 저장된다.

## 12. 개발 순서

### Phase 1: UI 프로토타입

샘플 데이터 20~40명으로 화면과 사용 흐름을 만든다.

- 검색
- 상세 화면
- 관계 이동
- 주라인 목록
- 미연결 목록

### Phase 2: Supabase 연결

- 프로젝트 생성
- `members` 테이블 생성
- 로그인
- RLS 정책
- 읽기·추가·수정·숨기기

### Phase 3: CSV 이전

- 기본 회원 데이터 일괄 등록
- 중복 회원번호 검사
- 상위 회원 자동 연결
- 확인 필요 목록 생성

### Phase 4: 실제 사용 테스트

엄마가 직접 다음 작업을 해본다.

- 회원번호 검색
- 닉네임 검색
- 상위 회원 이동
- 신규 회원 추가
- 잘못된 정보 수정
- 회원 숨기기
- 미연결 회원 연결

### Phase 5: 선택 기능 결정

- 최근 검색
- 이전·다음 회원
- 계보도 전체 보기
- 즐겨찾기 세분화

## 13. MVP 시작 조건

전체 데이터 정리가 끝날 필요는 없다.

개발 시작에 필요한 최소 조건:

- 샘플 회원 20~40명
- 정상 연결 사례
- 닉네임이 있는 사례
- 상위 회원을 모르는 사례
- 좌우가 없는 사례
- 탈퇴 회원 사례
- 주라인 회원 사례
- 한국·브라질 전화번호 사례

현재 샘플 CSV는 UI와 데이터 구조를 만들기 시작하기에 충분하다. 닉네임 해독과 전체 관계 연결은 앱 개발 이후 `미연결 회원` 화면을 이용해 계속 진행한다.
