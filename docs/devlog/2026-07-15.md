# 2026-07-15 Devlog — 회원 계보 검색 앱 MVP

## 작업 개요

엄마가 관리하던 회원 명단 시트를 바탕으로, 회원번호·이름·닉네임·전화번호를 검색하고 회원의 소속과 계보 관계를 확인할 수 있는 내부용 앱 MVP를 만들었다.

기존 시트의 실제 목적은 단순한 명단 보관이 아니라 다음과 같았다.

- 회원번호의 주인이 누구인지 확인
- 해당 회원이 어느 주요 사업자 조직에 속하는지 확인
- 좌·우 위치 확인
- 연결된 상위·하위 회원으로 이동
- 관심 회원과 주요 사업자를 빠르게 찾기

기존 시트는 약 1,400행 규모이며, 여러 사이트에서 복사한 자료가 섞여 있어 표기·닉네임·전화번호·관계 정보가 일관되지 않았다.

## 데이터 구조 정리

Supabase 이전을 고려해 다음 열을 사용하기로 했다.

```text
member_number
name
nickname
is_anchor_member
is_favorite
sponsor_name_raw
side
birth_date
phone
country_code
cpf
notes
member_status
is_hidden
```

주요 원칙:

- `name`: 회사 등록 이름
- `nickname`: 엄마가 실제로 사용하는 한국어 이름
- `member_number`, `phone`: 숫자로만 적되 text 타입 유지
- `country_code`: `KR`, `BR`, `US`, `MX`, `CO`, `XX`
- `XX`: 글로벌 회원
- 빈 국가 코드: 국가 미확인
- `member_status`: `active`, `withdrawn`, `review`
- 앱에서는 삭제하지 않고 `is_hidden`으로 숨김 처리
- 실제 삭제는 Supabase Dashboard에서만 수행

기존 `is_mainline`은 의미가 달라 다음 두 필드로 나누기로 했다.

```text
is_anchor_member
is_favorite
```

- `is_anchor_member`: 계보를 추적할 때 기준점이 되는 주요 사업자
- `is_favorite`: 엄마가 자주 확인하는 관심 회원

## 관계 데이터에 대한 발견

기존 `sponsor_name_raw`는 실제 직계 부모가 아니라, 엄마가 기억하는 가장 가까운 조상이나 주요 사업자를 적어둔 경우가 많았다.

예:

```text
테스트 소속/좌
테스트 알파/좌
공동 사용자 A
공동 사용자 B
```

따라서 현재 데이터만으로는 모든 회원의 정확한 직계 계보를 계산할 수 없다.

초기 앱에서는 이를 오류로 취급하지 않고 다음처럼 표시하기로 했다.

```text
소속
테스트 알파 · 좌
```

직계 관계가 정확히 연결된 경우에만 별도로 표시한다.

```text
바로 위 회원
김OO
```

향후 신규 회원 입력에서는 상위 회원 이름을 직접 쓰지 않고, 이름·닉네임·회원번호 검색 결과에서 기존 회원을 선택하도록 한다.

## 부부 공동 계정 처리

한 공동 계정은 회사 등록 이름과 실제 업무에서 쓰는 두 사용자 이름이 달라, 기존 시트에서 서로 다른 이름이 혼용되고 있었다.

현재 닉네임이 두 사용자 이름을 합친 값으로 저장되어 있어, 둘 중 하나만 적힌 관계는 자동 연결되지 않는 문제를 확인했다.

다음과 같이 공동 사용 이름을 별도 필드로 두는 방식을 검토했다.

```text
name: Demo Shared Account
nickname: 공동 사용자 A
co_user_name: 공동 사용자 B
```

검색과 관계 연결에서는 회사 등록 이름, 닉네임, 공동 사용 이름, 회원번호를 모두 검색한다.

## MVP UI 확정

가격표 앱의 디자인을 참고해 다음 화면 구조를 확정했다.

### 메인 화면

```text
회원 계보 찾기
[이름·닉네임·회원번호·전화번호 검색]

[전체] [주요 사업자] [관심 회원]
```

접힌 카드에는 최소 정보만 표시한다.

```text
Demo Member With A Long Name
테스트 소속 · 10000001 · XX
```

긴 이름은 한 줄 말줄임 처리한다.

### 펼친 카드

```text
전체 이름
닉네임
회원번호 · 국가

[주요 사업자] [관심 회원] [탈퇴] [확인 필요]

소속
테스트 알파 · 좌

바로 위 회원
김OO

생년월일 · 만 나이
전화번호
CPF
메모

연결된 회원
좌 목록
우 목록

[수정] [숨기기]
```

- 생년월일 기준 만 나이 자동 계산
- CPF는 `000.000.000-00` 형식으로 표시
- 전화번호는 국가별로 가능한 범위에서 형식화
- 연결된 회원은 회원번호 오름차순 정렬
- 관리 기능은 상단 `관리` 메뉴 안에 배치
- 앱 내 실제 삭제 기능 없음

## CSV 기반 MVP 제작

Supabase 연결 전에 CSV만으로 실제 화면과 동작을 테스트할 수 있도록 브라우저 MVP를 제작했다.

적용 기능:

- 회원번호·이름·닉네임·전화번호 통합 검색
- 정확한 회원번호 검색 시 카드 자동 펼침
- 전체·주요 사업자·관심 회원 필터
- 긴 이름 말줄임
- 카드 펼침형 상세정보
- 파스텔 상태 배지
- 소속 회원 클릭 이동
- 주요 사업자 경로 자동 표시
- 만 나이 자동 계산
- 국가별 전화번호 및 CPF 표시
- 좌·우 연결 회원 목록
- 회원 추가·수정·숨기기
- 숨긴 회원 복원
- CSV 교체·내보내기
- localStorage 임시 저장
- 실제 삭제 기능 없음

Supabase 운영 데이터는 시트 정리가 완료된 뒤 최초 일괄 업로드하기로 했다. 개발 중에는 CSV를 사용해 UI와 사용 흐름을 확정하고, 운영 DB에 임시 데이터와 실제 데이터를 반복해서 덮어쓰는 작업은 피한다.

## 배포 계획

정식 주소:

```text
member.nangok.app
```

진행 순서:

```text

```

## 오늘의 핵심 발견

이 앱의 핵심은 전체 계보를 완벽하게 재현하는 것이 아니라, 회원번호 하나만 알고 있을 때 그 사람이 어느 주요 사업자 조직에 속하는지 빠르게 확인하는 것이다.

기존에는 엄마가 모바일 계보도를 직접 스크롤하며 회원번호를 찾았지만, 앱에서는 회원번호 검색만으로 소속과 연결 정보를 확인할 수 있다.

데이터가 완벽하지 않아도 UI 개발은 진행할 수 있으며, 직계 관계가 불완전한 회원은 기존 조상 연결을 소속 단서로 활용하고 실제 사용 과정에서 점진적으로 보완하기로 했다.
