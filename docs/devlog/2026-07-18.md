# 2026-07-18 ngmembers Development Log

## Work Summary

회원앱을 Supabase 실서비스로 전환하기 전에 Phase 0 기준선을 확정했다. 기능이나 UI를 변경하지 않고 현재 구조·데이터 흐름·빌드·개인정보 노출·최종 CSV·웹폰트를 점검했으며, 재현 가능한 결과를 `docs/reports/phase-0-baseline.md`에 기록했다.

점검 과정에서 명령은 성공하지만 최신 `src`가 실제 프로덕션 번들에 포함되지 않는 빌드 entry 문제와, 실제 회원 데이터가 public GitHub 저장소 및 GitHub Pages 번들에 포함된 보안 문제를 확인했다.

이후 기존 원격 저장소를 삭제하고 같은 이름의 새 저장소를 만든 상황을 기준으로, 구현 전에 Phase 1의 안전 기준·작업 순서·완료 조건을 정리했다. 초보 사용자가 전체 개발 순서를 빠르게 파악할 수 있도록 루트에 간단한 `ROADMAP.md`도 추가했다.

J님 승인 후 Phase 1 로컬 구현을 진행했다. 실제 회원 seed·CSV와 과거 번들을 제거하고 정상 Vite entry를 복구했으며, 개인정보 없는 fixture 위에 Supabase client, Google Auth, 고정 workspace Admin 권한 gate, members schema와 RLS를 추가했다. 공용 schema 차이를 반영한 SQL 재적용, 두 Admin 동작과 DELETE 차단, 안전한 새 Git 이력까지 확인해 Phase 1을 완료했다.

Phase 1 완료 후 Phase 2 실행 계획을 작성했다. 실제 CSV를 즉시 올리지 않고 source hash 고정, strict validation, 명시적 보정 승인, 원자적 service-role RPC import, 사후 검증과 제한된 rollback을 거쳐야 production 반영할 수 있도록 단계와 중단 조건을 구체화했다.

이어서 Phase 2의 로컬 도구와 일회성 DB import 경로를 구현했다. 최신 CSV는 기존 오류 6건이 모두 해소된 1,378행으로 확인됐지만 경고 승인은 아직 없으므로, private report와 빈 보정 manifest까지만 생성하고 prepared payload와 production import는 안전장치가 차단하도록 유지했다.

J님이 빈 필드는 실제 미등록 값으로, sponsor 미해결·복수 후보는 미연결로, CPF·전화번호 형식 의심값은 현재 원문으로 이전해 나중에 앱에서 수정하는 방침을 승인했다. 이 결정을 source hash에 묶어 기록한 뒤 deterministic prepared payload와 rollback UUID 목록을 생성하고 오프라인 dry-run까지 완료했다.

Phase 3에서는 권한 기준선과 생성 타입을 읽고 React 데이터 계층을 Supabase로 교체했다. UUID cursor 전체 로드, mapper, create/update/hide, `updated_at` 충돌 처리와 비동기 UI를 구현했으며 demo·localStorage 회원 저장·CSV 교체·복원 경로를 제거했다. Free 요금제에서 유료 Branching을 사용할 수 없고 로컬 Docker도 없는 상황을 J님과 확인한 뒤, 명시적 승인에 따라 Docker 검증을 생략하고 production 권한 migration을 적용했다. 적용 전후 데이터 불변, 최소 권한, 순환 차단 trigger와 Advisor 회귀 부재를 확인했다.

Phase 2–3 커밋 직후 GitHub Pages에서 흰 화면이 발생했다. 로컬 앱은 정상이고 배포본의 `#root`만 비어 있음을 재현한 결과, legacy Pages가 Vite build 결과가 아닌 저장소 루트의 `index.html`을 배포해 브라우저가 `src/main.tsx` 원본을 직접 요청하는 것이 원인이었다. Pages 배포를 GitHub Actions workflow로 전환해 검증된 `dist` artifact만 게시하도록 수정했다.

## Decisions

- 기존 원격과 일치하는 `3c67826`을 변경 전 기준 커밋으로 사용하고 의미 없는 빈 커밋은 만들지 않았다.
- Phase 0에서는 발견된 빌드·PII·CSV 문제를 수정하지 않고 분석 결과만 문서화했다. 이는 현재 UI를 보존하고 다음 단계의 승인 범위를 분명히 하기 위해서다.
- 최종 CSV 원본과 실제 개인정보 값은 저장소 보고서에 복사하지 않았다. 문제 식별에는 CSV 행 번호와 필드 유형만 사용했다.
- sponsor 검증은 계획대로 이름·닉네임·회원번호의 정확 일치를 사용하고 공백과 대소문자만 정규화했다.
- public 저장소와 Pages의 PII 노출 차단을 Phase 1보다 먼저 결정해야 하는 최우선 사항으로 분류했다.
- 현재 index entry 문제를 해결하기 전에는 `npm run build` 성공을 최신 앱 배포 성공으로 간주하지 않기로 했다.
- entry 복구는 단독으로 먼저 배포하지 않고 실제 seed·과거 bundle 제거와 함께 수행하기로 했다. 정상 entry가 현재 seed를 다시 bundle하는 위험을 피하면서도 Phase 1 초반에 최신 build 기준선을 확보하기 위한 결정이다.
- 새 public 저장소에는 PII 검사에 통과한 안전한 첫 commit만 올리고, Phase 1 계획 문서 작성 단계에서는 구현이나 push를 수행하지 않기로 했다.
- J와 엄마는 앱 안에서 허용되는 작업이 같으므로 두 계정을 모두 `admin`으로 사용하기로 했다. 별도 `editor` 역할은 실질적인 권한 차이 없이 코드와 RLS 분기만 늘리므로 사용하지 않는다.
- 두 Admin 계정 모두 조회·추가·수정·숨기기만 할 수 있으며 DELETE 정책과 삭제 기능은 만들지 않기로 했다. 숨긴 항목의 복원·영구 삭제는 앱 밖에서 J가 Supabase Dashboard를 통해 처리한다.
- 기존 엄마용 앱 Supabase project를 재사용하고 ngmembers workspace ID는 `00000000-0000-0000-0000-000000000003`으로 정했다. 같은 project에 연결된 Google OAuth Client도 재사용하는 방향으로 확정했다.
- 별도 `app_users` 허용 목록을 중복 생성하지 않고 기존 `workspaces`와 `workspace_members`를 재사용하기로 했다. 두 계정은 이메일 문자열 대신 기존 Supabase Auth 사용자 UUID로 ngmembers 워크스페이스에 연결한다.
- 공유 project 호환성을 위해 `workspace_members.role` 전체를 Admin만 허용하는 check constraint는 만들지 않고, ngmembers 권한 helper에서만 `admin`을 요구하기로 했다.
- 실제 공용 `workspace_members`에는 활성 상태 컬럼이 없으므로 별도 컬럼을 자동 추가하지 않기로 했다. 기존 접근 모델을 보존해 멤버십 행의 존재를 활성 상태로 보고, 접근 회수는 해당 행 삭제로 처리한다.
- Phase 1에서는 실제 DB CRUD 전환을 앞당기지 않고 기존 UI를 가상 fixture와 Phase 1 전용 localStorage로 유지했다. 데이터 계층 전체 교체는 계획대로 Phase 3에 수행한다.
- 이전 Git seed를 메모리에서만 읽어 current source/dist와 대조하는 안전 검사기를 추가했다. 실제 marker 값이나 별도 개인정보 파일을 새 저장소에 기록하지 않으면서 잔존 여부를 재현 가능하게 확인하기 위한 결정이다.
- SQL은 기존 공용 schema를 자동 추측해 변경하지 않고 preflight에서 불일치를 중단하도록 했다. 기존 엄마용 앱의 테이블 구조를 모르는 상태에서 자동 migration으로 공유 기능을 깨뜨리지 않기 위해서다.
- DELETE 버튼 부재와 DB 권한 차단을 별도로 유지하기로 했다. UI는 실수를 막고, table grant와 RLS는 직접 REST 호출까지 차단하는 서로 다른 보안 계층이다.
- 새 공개 Git 이력은 기존 commit을 수정하는 방식이 아니라 현재 안전 tree를 부모 없는 root commit으로 만드는 방식을 선택했다. 원격에 PII 과거 object가 전달될 경로를 구조적으로 없애기 위해서다.
- 과거 object 정리 후에도 PII 대조 검사를 이어갈 수 있도록 ignore된 `scripts/pii-markers.local.txt`를 선택적으로 읽게 했다. 실제 marker를 Git에 복원하지 않으면서 Phase 2 이후 로컬 검증 자료를 사용할 수 있게 하기 위한 결정이다.
- Phase 2 production import는 브라우저용 CSV parser를 재사용하지 않고 strict TypeScript validator와 local-only prepared payload를 사용하기로 했다. 기존 parser는 잘못된 값을 기본값으로 바꾸는 등 운영 이전에 필요한 중단 조건보다 느슨하기 때문이다.
- 기본 행 insert와 관계 update는 service-role 전용 일회성 RPC 한 호출로 묶기로 했다. 여러 REST 요청의 중간 실패로 일부 회원만 남는 상황을 피하고 PostgreSQL transaction의 원자성을 사용하기 위해서다.
- `members`에 영구 `import_batch_id` 컬럼을 추가하지 않고 prepared UUID manifest로 최초 import를 식별하기로 했다. 일회성 이전 때문에 운영 schema를 늘리지 않으면서도 제한된 rollback 대상을 고정할 수 있다.
- sponsor 미해결·복수 후보와 형식 경고는 자동 추측하지 않고 J님 승인 후 `NULL`과 private report로 남기기로 했다.
- 보정 manifest는 source SHA-256에 묶고 승인자·시각·근거를 요구하기로 했다. 계획서 승인과 실제 값 보정·경고 수용을 구분해 최신 CSV가 바뀌거나 승인 기록이 비어 있으면 prepared payload를 만들지 않기 위해서다.
- prepared UUID는 무작위로 매번 바꾸지 않고 source hash와 CSV 행 번호에서 결정적으로 생성하기로 했다. 같은 승인 입력에서 payload hash와 rollback 대상을 재현할 수 있게 하기 위한 결정이다.
- production apply에는 계획서의 네 hash/count flag 외에 project ref 재입력과 backup 확인 flag도 요구하기로 했다. 공유 project 오선택과 복구 지점 미확인 상태의 쓰기를 client 단계에서 한 번 더 막기 위해서다.
- 최신 CSV의 오류 0건은 값 보정 승인 완료로 확대 해석하지 않기로 했다. 경고 80건의 범주별 승인 전에는 actual prepared payload와 DB write를 만들지 않는다.
- J님은 CPF·전화번호가 schema상 `text`라 형식 의심값도 DB 입력을 막지 않는다는 점을 확인한 뒤, 현재 원문을 이전하고 사용자가 나중에 화면에서 수정하는 방식을 승인했다. 실제 미등록 회원번호·이름·국가는 `NULL`, sponsor 미해결·복수 후보는 추측 없이 미연결로 유지한다.
- Production preflight와 apply를 분리하고 preflight는 읽기 전용 count만 수행하기로 했다. 기존 행이 있으면 알려진 테스트 marker인지 count로만 확인하되, script가 자동 삭제하거나 병합하지 않도록 해 실제 데이터 오삭제를 방지한다.
- 일회성 RPC의 접근 제어는 API key 내부 형식이 아니라 PostgreSQL 함수 EXECUTE 권한을 기준으로 삼기로 했다. 새 secret key는 JWT가 아니므로 `request.jwt.claim.role`이 없지만 `service_role` DB 권한으로 실행되며, `public`·`anon`·`authenticated`의 실행 권한 회수가 실제 보안 경계다.
- Phase 3은 1,378명 전체 graph를 브라우저 메모리에 로드해 기존 검색·계보 계산을 유지하되, 단일 select의 row limit를 피하도록 UUID primary key cursor와 500행 batch를 사용하기로 했다.
- 두 Admin의 동시 수정은 `updated_at` 비교 조건으로 오래된 update를 차단하고, Realtime은 focus refresh와 conflict detection의 실사용 결과를 본 뒤 별도 단계로 미루기로 했다.
- 운영 UI에서는 CSV import/export·샘플 초기화·복원·영구 삭제를 제거하고 create/update/hide만 DB에 연결하기로 했다.
- production 전체 graph는 클라이언트가 임의 기본값으로 보정하지 않고 mapper 오류로 로드를 중단하기로 했다. 일부 잘못된 row만 노출하는 것보다 데이터 문제를 명시적으로 확인하는 편이 안전하기 때문이다.
- production 권한 migration과 가짜 CRUD는 로컬 Docker나 승인된 development branch 없이 production에 시험 적용하지 않기로 했다. 코드·정적 검사를 완료하되 DB 변경과 실제 계정 검증은 J님 승인 후 분리해 진행한다.
- 공유 project Advisor의 기존 다른 앱 경고는 Phase 3 범위에서 수정하지 않기로 했다. ngmembers migration 적용 전후 cache key를 비교해 새 경고만 회귀로 판단한다.
- 관계 cycle 검사는 클라이언트 사전 검사와 DB 재귀 trigger를 함께 사용하고, DB trigger의 동시 transaction은 하나의 advisory lock key로 직렬화하기로 했다. 두 Admin 규모에서 관계 write 빈도보다 무결성 보장이 더 중요하기 때문이다.
- Supabase Free 요금제에서는 preview Branching을 사용할 수 없고 Docker 설치는 필수 운영 조건이 아니므로, J님 승인으로 두 검증 경로를 생략했다. 대신 production 적용 직전 읽기 전용 기준값을 고정하고, 행을 변경하지 않는 단일 migration 적용 후 동일 집계·권한·함수 속성·롤백 순환 테스트·Advisor를 연속 검증하는 방식으로 위험을 통제했다.
- Vite source와 build artifact를 같은 것으로 취급하지 않기로 했다. GitHub Pages는 저장소 root의 TypeScript/TSX를 직접 게시하지 않고, Actions에서 검증을 통과한 `dist`만 배포한다.
