## 1. 결론

현재 프로젝트는 TypeScript 검사와 Vite 명령 자체는 성공하지만, 실제 프로덕션 번들은 최신 `src`가 아니라 저장소에 이미 들어 있던 과거 완성 번들을 다시 입력으로 사용한다. 따라서 현재 상태를 단순히 “빌드 정상”으로 판정할 수 없다.

또한 실제 회원 개인정보가 Git이 추적하는 seed JSON, 샘플 CSV, 완성 JS 번들에 포함되어 있으며 원격 저장소가 `public`이고 GitHub Pages도 HTTP 200으로 공개되어 있다. 공개 페이지의 JS 번들에서 seed 회원의 회원번호·이름·생년월일·전화번호 포함 여부를 값 노출 없이 확인했으며 모두 발견되었다. 이 노출 차단은 Phase 1보다 먼저 처리해야 한다.

최종 `ngmembers.csv`는 1,378행과 필수 14개 헤더를 정상적으로 읽었지만, 엄격한 Phase 2 규칙 기준으로 오류 6행이 있어 아직 import할 수 없다. 경고는 84건이며 82행에 분포한다.

### 빌드 기준선 문제

루트 `index.html`은 Vite의 정상 React entry인 `/src/main.tsx`를 참조하지 않고 다음 과거 산출물을 직접 참조한다.

```text
./assets/index-DhHxHh3o.js
./assets/index-C3QC9xHP.css
```

`tsc`는 최신 `src`를 타입 검사하지만, Vite는 위 과거 JS/CSS를 입력으로 다시 번들링한다. 다음 문구 대조로 이 상태를 확인했다.

| 확인 문구 | 최신 `src` | 기존 root JS | 새 `dist` JS |
| --- | --- | --- | --- |
| `스폰서 선택` | 있음 | 없음 | 없음 |
| `스폰서 연결 정리` | 있음 | 없음 | 없음 |
| `소속 회원 선택` | 없음 | 있음 | 있음 |

따라서 새 `dist`도 최신 React UI가 아니다. 이후 구현을 시작하기 전에 `index.html`의 Vite entry를 복구하고, 최신 `src`가 실제로 번들에 포함되는 기준선을 다시 만들어야 한다.

## 4. 현재 데이터 흐름

```text
src/data/seedMembers.json
        │ 최초 실행 또는 저장 오류
        ▼
src/lib/storage.ts
        │ loadMembers()
        ▼
App의 members 상태
        │
        ├─ 검색·필터·정렬
        ├─ 관계 인덱스와 데이터 오류 계산
        ├─ 추가·수정·숨김·복원
        ├─ CSV 전체 교체·내보내기
        └─ 모든 변경마다 saveMembers()
                 ▼
localStorage: nangok-member-mvp-v1
```

- 서버 통신, 인증, 공유 DB는 없다.
- 브라우저별 `localStorage`가 유일한 수정본이므로 다른 브라우저·기기와 공유되지 않는다.
- 저장소 접근이 실패하거나 payload version이 맞지 않으면 seed로 되돌아간다.
- 첫 렌더 후 seed를 포함한 전체 회원 배열이 `localStorage`에도 저장된다.
- CSV import는 최소 헤더 두 개만 확인하고 현재 브라우저 데이터를 전부 교체한다.
- 현재 import 변환은 잘못된 상태를 `review`, 잘못된 boolean을 `false`, 잘못된 side를 `null`로 완화하므로 Phase 2의 중단형 검증기로 재사용할 수 없다.
- 현재 관계 해석은 악센트·문장부호·공백까지 제거하는 `normalizeSearch`를 사용한다. Phase 2의 sponsor 연결 규칙은 공백·대소문자만 정규화하는 정확 일치이므로 별도 구현이 필요하다.

## 5. 현재 UI 동작

### 메인 목록

- 이름·닉네임·회원번호·전화번호를 하나의 검색창에서 검색한다.
- 정확히 한 명과 일치하면 해당 카드를 자동으로 펼친다.
- 검색 우선순위는 정확한 회원번호, 닉네임, 이름, 전화번호 순이며 이후 접두·포함 검색을 적용한다.
- `전체`, `주요 사업자`, `관심 회원` 필터와 각 건수를 표시한다.
- 숨긴 회원은 메인 목록, 선택기, 연결 회원 목록에서 제외한다.
- 회원번호를 숫자 인식 locale 정렬로 오름차순 정렬한다.

### 회원 카드

- 접힌 카드에서 이름은 한 줄 말줄임, 펼친 카드에서는 전체 이름을 표시한다.
- 닉네임·회원번호·국가 코드는 값이 있을 때만 보인다. 회원번호가 없으면 현재는 `회원번호 미등록`을 표시하지 않고 자리를 생략한다.
- 펼친 카드에서 주요 사업자·관심 회원·탈퇴·확인 필요 배지를 표시한다.
- 현재는 `status=review`뿐 아니라 자동 데이터 이슈가 하나라도 있으면 `확인 필요` 배지가 붙는다.
- 스폰서, 주요 사업자 경로인 `라인`, 정확히 입력된 바로 위 회원, 개인정보, 연결된 회원을 표시한다.
- 연결 회원은 좌·우·위치 미확인으로 나누고 각 그룹 8명 이후를 접는다.
- 수정과 숨기기는 가능하며 삭제 기능은 없다.

### 추가·수정

- 회원번호와 가입 이름을 모두 필수로 요구한다.
- 회원번호는 숫자만 입력되고 중복 저장을 막는다.
- 닉네임은 한글 두 글자 이상만 허용한다.
- 소속과 바로 위 회원은 검색 선택기로 고른다.
- 자기 자신 연결과 관계 순환을 막는다.
- 생년월일·전화번호·CPF는 숫자만 남겨 저장한다.

### 관리

- 데이터 확인, 탈퇴 회원, 숨긴 회원 복원, 스폰서 미해결 목록을 제공한다.
- 역할 구분이나 로그인은 없다.
- CSV 전체 교체, 전체 내보내기, seed 초기화가 남아 있다.

## 10. 웹폰트 상태

### 현재 연결 방식

- 루트 및 빌드된 `index.html`이 jsDelivr의 Pretendard 1.3.9 variable CSS를 외부 `<link rel="stylesheet">`로 로드한다.
- `src/styles.css`의 body stack은 `Inter`, `"Pretendard Variable"`, `Pretendard`, `system-ui`, `sans-serif` 순서다.
- 프로젝트에는 `.woff`, `.woff2`, `.ttf`, `.otf` 파일이 없다.
- source CSS와 빌드 CSS에 자체 `@font-face`가 없다.
- `dist/assets`에도 폰트 파일이 생성되지 않는다.
- 점검 시점의 CDN CSS는 HTTP 200, 실제 WOFF2는 HTTP 200과 `Access-Control-Allow-Origin: *`로 응답했다.
- Pretendard 공식 문서도 현재 사용 중인 jsDelivr 링크와 `"Pretendard Variable"` family 이름을 안내한다: https://github.com/orioncactus/pretendard

### 배포에서 적용되지 않을 수 있는 이유

1. 폰트가 배포 산출물에 포함되지 않아 CDN, 네트워크, CSP, 차단 확장 프로그램에 전적으로 의존한다.
2. `Inter`가 stack의 첫 번째지만 프로젝트에서 로드하지 않는다. 기기에 Inter가 설치된 환경과 그렇지 않은 환경에서 라틴 글자 선택이 달라질 수 있다.
3. 루트 `index.html`이 해시된 과거 산출물을 직접 가리키므로 새 CSS·폰트 변경을 해도 최신 React 빌드와 함께 배포되지 않을 수 있다.
4. 새 빌드의 `dist/index.html`은 새 JS 해시를 가리킨다. `dist` 전체가 아니라 일부 파일만 올리면 index와 assets 해시가 어긋난다.

실서비스에서 외부 CDN 의존을 유지할지, 확정된 WOFF2를 `src/assets/fonts`로 넣어 Vite가 해시·복사하도록 할지 결정해야 한다. 실제 브라우저의 Network 200, Computed Style, 모바일 굵기 검사는 build entry와 배포본을 바로잡은 뒤 다시 수행해야 한다.
